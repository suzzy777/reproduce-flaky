With patch005, nothing is really changed, I just added back the original implementation of {{testQueueNameContainingNBWhitespace}} to {{TestFairScheduler}} to be able to check how this test behaves while Jenkins executes the tests.
I would expect that to fail as well as my previous (004) patch only contains test code changes so the code should work the same as before. 

Actually it turned out that the {{"file.encoding"}} system property can be different on linux and mac.
I just ran the {{./start-build-env.sh}} script from the root dir of Hadoop and executed my testcase with: 

{code:java}
mvn -Dtest=TestAllocationFileLoaderService#testQueueNameContainingNBWhitespace test
{code}

I temporarily redefined my testcase as: 


{code:java}
@Test (expected = AllocationConfigurationException.class)
  public void testQueueNameContainingNBWhitespace() throws Exception {
    Configuration conf = new Configuration();
    conf.set(FairSchedulerConfiguration.ALLOCATION_FILE, ALLOC_FILE);

    AllocationFileWriter.create()
        .addQueue(new AllocationFileQueue.Builder("\u00a0").build())
        .writeToFile(ALLOC_FILE);
    System.out.println("sysprop: file.encoding::::" + System.getProperty("file.encoding"));
    System.out.println("ALLOC_FILE: " + ALLOC_FILE);
    AllocationFileLoaderService allocLoader = new AllocationFileLoaderService();
    allocLoader.init(conf);
    ReloadListener confHolder = new ReloadListener();
    allocLoader.setReloadListener(confHolder);
    allocLoader.reloadAllocations();
  }
{code}


It prints the following: 

{code:java}
sysprop: file.encoding::::ANSI_X3.4-1968
default charset: US-ASCII 
{code}


US-ASCII was used as a default charset on the JVM running on Linux (the docker container itself) so I added UTF-8 as character encoding parameter to the constructor of the {{PrintWriter}} in {{AllocationFileWriter}}
With this update, the testcase finally passes. 
Expecting same results from Jenkins.
